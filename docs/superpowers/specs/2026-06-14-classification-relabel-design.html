Design Specification

In-Session Classification Relabel (Step 1.5)

Project: public-mutual-funds-analyzer Skill: fund-screener Date: 2026-06-14 Status: Approved design — pending implementation plan

1. Problem

extract_mfr.py assigns four classification fields per fund using keyword/substring matching: asset_class, geography, objective_class, and phs_fund_type. This ladder-of-if-statements cannot understand a fund — it only matches anticipated tokens — so hard cases silently fall through to defaults ("Other", "", or "Malaysia").

Measured against the tracked mfr_results.json (171 funds), the keyword classifier was wrong on 12 of 13 genuinely ambiguous funds — all silently, with no error:

  • 8/8 wrong on asset_class — all dumped to "Other" (e.g. Greater China, ASEAN, Islamic ESG equity funds; and a money-market fund).
  • 4/5 wrong on geography — defaulted to "Malaysia" (e.g. China Ittikal, Asia Ittikal, regional/global funds).

Two are outright code bugs: 'MONEY MARKET' (with a space) fails to match MONEYMARKET (line 617), and the ITTIKAL→Malaysia rule runs before the CHINA/ASIA rules (line 650), so a China fund is labelled Malaysia.

2. Goal & non-goals

Goal. Raise the accuracy of the four fuzzy classification fields by having Claude relabel only the funds the keyword classifier punted on, during the normal skill run — no API key, no console spend, no change to the user's routine (download MFR → invoke fund-screener).

Non-goalWhy
Touching numeric / transcribed fields (performance, ytd, weighted_alpha, asset_allocation, top5_holdings, top5_sectors)These are deterministic and authoritative; an LLM could only add silent drift to numbers and proper nouns shown to clients.
Editing extract_mfr.py (including the two known bugs)Decision locked: LLM-relabel-only. The failure trigger already catches both bug cases (see §4).
Re-classifying all 171 fundsDecision locked: failures-only. A confident keyword result is never second-guessed.
A standalone script / API pathNot needed while a Claude Code session drives the run. Reserved for a future fully-unattended pipeline.

3. Architecture

A new instruction block — Step 1.5 — in fund-screener-skill/SKILL.md, executed by Claude between Step 1 (extract_mfr.py) and Step 3 (build_sheet_data.py). No new scripts. No Python changes. It is a read-modify-write on mfr_results.json. 

Step 1   extract_mfr.py        → mfr_results.json   (keyword classification)
         ── Step 1.5  relabel pass (Claude, in-session) ──  rewrites mfr_results.json
Step 2   fetch_ath.py          → ath_results.json
Step 3   build_sheet_data.py   → master_funds.csv   (reads corrected JSON, unchanged)
Step 4   build_xlsx.py         → workbook           (unchanged)

Step 1.5 is independent of the ATH fetch, so its order relative to Step 2 does not matter. Because Steps 3–4 read the classification fields straight out of the JSON, correcting them in the JSON requires zero downstream changes.

4. Trigger — per-field low-confidence test

A fund's field is relabeled only when its deterministic value is low-confidence:

FieldRelabel when…
asset_class== "Other"
objective_class== ""
phs_fund_type== ""
geography== "Malaysia" and "malaysia" not in lipper_class (lowercased)
Why this catches the bugs without touching extract_mfr.py. PIMMF-A (MONEYMARKET bug) surfaces as asset_class == "Other". The ITTIKAL/regional geography bugs (PCIF, PAIF, PRSEC, PeISMF) all have geography == "Malaysia" while their lipper_class reads China / Asia / Global — so the geography rule flags them. No keyword logic is re-implemented.

The geography rule also flags empty-lipper Malaysia funds (empty string trivially lacks "malaysia"). That is intentional: empty lipper = low signal = worth a look. A fund that is genuinely Malaysian simply gets confirmed — a harmless no-op.

5. Behaviour per flagged fund

  1. Read the fund's name, lipper_class, and objective_text — all already present in mfr_results.json.
  2. Choose the corrected value from the field's bounded enum (§6). No free text.
  3. Write the corrected value back into the fund object, and stamp a sibling <field>_source: "llm-relabel" for audit.
  4. Leave every other field byte-for-byte unchanged.

6. Bounded enums

Values must come from the vocabulary the Excel columns already expect, so nothing downstream breaks.

FieldAllowed values
asset_classEquity - Malaysia · Equity - Foreign · Bond / Fixed Income · Bond / Sukuk · Mixed Asset · Balanced · Money Market
geographyMalaysia · Greater China · ASEAN · Asia Pacific · Asia · Global · United States · India · Indonesia · Japan · Australia · Vietnam · Singapore
objective_classCapital Growth · Income · Capital Growth + Income
phs_fund_typeEquity · Fixed Income · Mixed Asset / Balanced · Money Market · Fund of Funds (one combined value — not split, unlike asset_class)

7. Error handling & edge cases

  • Unresolvable field (e.g. empty objective_text): keep the original keyword value and do not set _source. Never blank a field.
  • No enum fits (e.g. PeEMAS, a gold/commodity fund-of-funds): leave the value as-is rather than force a wrong enum. A genuine "Other" is allowed to remain.
  • Never invent an out-of-enum value, and never modify a non-targeted field.

8. Reproducibility

Bounded enums plus explicit rules keep output stable in practice. Step 1 regenerates mfr_results.json on every run, so the relabel re-runs every run. Both the proposal and the FundMaster workbook derive from the same post-relabel JSON within a run, so tests/test_proposal_validation.py stays internally consistent. The _source flag provides a clean audit trail.

Accepted tradeoff. Because the relabel is model-driven, a label could occasionally flap month-to-month on a borderline fund. This is cosmetic (a category label, never a number or a gate input) and traceable via _source.

10. Testing & verification

  • Sanity: after Step 1.5, asset_class == "Other" count ≈ 0 (legitimate commodity cases excepted); no targeted field left empty except where objective_text is absent.
  • Enum conformance: every relabeled value is within its §6 enum.
  • Non-interference: numeric/holdings/sector/allocation fields identical to the Step 1 output (diff check).
  • Existing eval: tests/test_proposal_validation.py continues to pass against a regenerated proposal — the KNOWN_* pin sets stay empty.

11. Acceptance criteria

  • Invoking fund-screener on the current MFR set yields asset_class == "Other" ≈ 0 and no spurious "Malaysia" on China/Asia/Global funds.
  • PIMMF-A → Money Market; PCIF → Greater China; PAIF → Asia; PGCF/PASGF → Equity – Foreign.
  • No numeric/holdings/sector value changed by Step 1.5.
  • User routine unchanged: download MFR, invoke skill.
