4aAllow derived-but-consistent numbers. Prose may introduce numbers not literally in a slot (e.g. an average) as long as they're consistent. The judge checks consistency, not novelty — forbidding new numbers would gut synthesis prose. See §4a for the precise definition.
4bHolistic per slot-instance. The per-claim enumeration + the rubric instruction "check EACH numeric claim independently before answering" are the blind-spot mitigations.

§4a — What "consistent" means

The judge's question about any number in the prose is not "is this number literally in the figure set?" (that would reject legitimate synthesis such as averages) but "can I trace this number back to the figures, and does it clash with any of them?" A number is consistent if it falls into one of three cases:

  1. Restatement — the number is one of the figures. PAPFF's 3yr alpha is 2.1%; prose says "outperformed by 2.1%." Trivially consistent.
  2. Correct derivation — not in the figure set but a valid arithmetic/logical result of it:
    • average — "its three equity funds average ~1.9% alpha" when the figures are 2.1, 1.8, 1.8 → mean 1.9 ✓
    • rounding — figure 2.13%, prose "about 2.1%" ✓
    • rank / superlative — "the top performer of the three" when it genuinely holds the highest alpha ✓
    • direction — "outperformed its benchmark" when alpha > 0 ✓
  3. Non-contradiction band — an approximate/soft number the judge cannot derive exactly but that conflicts with nothing. "A modest ~2% edge" against an actual 2.1% ✓.

A number is inconsistent (the complement) when it either:

  • contradicts a figure — "a 9% edge" when alpha is 2.1%; or
  • cannot be traced to any figure — fabricated with no derivation path (prose cites a Sharpe ratio or an "8% annual return" when state holds no such number). Unverifiable-as-stated-fact is treated as inconsistent, because it cannot be confirmed and reads as a hard claim.
The wrongly-derived trap. The subtle boundary — and the reason the seeded-bad fixtures exist — is prose that claims a derivation the figures don't support ("average 3%" when the true mean is 1.9%). This is case-2-shaped but inconsistent, because the claimed relationship to the figures is false. The judge must catch it, not wave it through as "a new number."
Keep the arithmetic in Python, not the judge. Asking an LLM to verify a derivation ("is 4.1 the mean of 4.0/4.1/4.2?") is the least reliable thing to delegate to it — LLMs are weak arithmetic checkers, an industry-recognized failure mode. So figures_extractor pre-computes the derivations a consultant is likely to state — per-sleeve and portfolio weighted-average alpha, weighted CFS, benchmark beat count/share — directly into the figure set (§6.1, §9). The judge then usually checks a prose number for membership / non-contradiction against an enriched figure set rather than re-deriving it; fresh arithmetic by the judge shrinks to the residual long tail. This is the deliberate hybrid — deterministic math + LLM semantic mapping — that the pure-LLM alternative would get wrong.

§6 Architecture — the four units

1 · figures_extractor pure · Python

Purpose: expose only the judge-relevant deterministic numbers as a flat dict, ready to inject as ground truth.
Interface: extract_figures(state) -> dict — pure, no I/O.
Fields: per-fund CFS+rank, weighted alpha, allocation %, exposure %; portfolio-level weighted aggregates.
Pre-computes the likely derivations into the figure set — per-sleeve + portfolio weighted-avg alpha, weighted CFS, benchmark beat count/share — so the judge checks membership, not arithmetic. Adding a derived field is cheaper/safer than trusting the judge to compute it.
Home: engine-side (consultant_engine/evals/) or evals/prose_numbers/; unit-tested.

2 · Fixture corpus frozen JSON

Purpose: freeze (prose, figures, verdict) triples that prove the judge before it gates.
Layout: evals/prose_numbers/fixtures/*.json.
Categories: good · seeded-bad-single · seeded-bad-buried.
The good set must include adversarial-but-correct derived cases (rounding, subset/weighted averages, superlatives per 4a) — these measure the judge's false-positive rate / precision, not just recall on planted errors.
Coverage: why.*/watch.*/macro.impact, multiple funds.

3 · Judge prompt + Promptfoo config Claude judge

Files: promptfooconfig.yaml + rubric prompt.
Output contract: strict JSON with a per-claim verdict list reduced to the slot verdict (see §6.3 below).
Assertion: a javascript/python parse of that JSON (preferred over a bare llm-rubric) that the reduced verdict equals the fixture's expect and, for seeded-bad-*, that the planted claim is the flagged one.
Model: a pinned Anthropic Claude (Sonnet-class) id. Multi-sample repeats surface non-determinism.

4 · CI gate GitHub Actions

Purpose: green only when every verdict matches its fixture's expect within a stated threshold.
Threshold: catch 100% of seeded-bad-*, pass 100% of good, across N samples.
A missed buried-error fails the suite red.

§6.3 — Judge output contract (per-claim verdict list)

Strict JSON: the judge enumerates every numeric claim with a per-claim verdict, then code reduces those to the slot-level boolean.

{
  "claims": [
    { "text": "outperformed by 2.1%",        "verdict": "entailed" },
    { "text": "top performer of the three",    "verdict": "entailed" },
    { "text": "a 9% edge",                      "verdict": "contradicted" }
  ],
  "entailed": false,
  "offending_sentence": "…a 9% edge over its benchmark."
}

Per-claim verdict ∈ {entailed, contradicted, underivable}. entailed is the reduction — false if any claim is contradicted or underivable; offending_sentence is the sentence of the first failing claim. Enumerating every numeric claim before reducing is the structural mitigation for the buried-error blind spot (§7) — it forces a verdict on each number rather than one holistic boolean reachable by reading only the first plausible claim. Stronger than the rubric instruction alone, and it keeps the no-regex property: the judge still finds the claims itself. The assertion is a javascript/python parse (preferred over a bare llm-rubric) that also checks the planted claim is the flagged one.

§7 Known blind spot + the buried-error experiment

The blind spot. A single slot's prose often carries several numbers. A holistic judge (policy 4b) could rubber-stamp entailed:true when one number is wrong but "buried among several" correct ones. This is the eval-side face of the well-documented lost-in-the-middle weakness — a judge reasoning over a list tends to miss the one bad item among many good ones; collapsing a multi-number slot to a single boolean invites it (the judge settles on a first plausible reading and stops).

Three layers of defense, structural first:

  1. Per-claim verdict list (structural, in the output contract, §6.3) — the primary mitigation. The judge must emit a verdict for every numeric claim before the boolean is reduced from that list in code; it cannot return entailed:true without a verdict on the buried number. A schema constraint, not a plea.
  2. Rubric instruction — the prompt still says "check EACH numeric claim independently before answering," reinforcing (1); per the industry parity result, the rubric must be as detailed as an atomic-decomposition prompt to earn holistic parity.
  3. The seeded-bad-buried fixtures — the acceptance crux — a slot with multiple correct numbers plus exactly one planted wrong one, proving the two layers above actually work.
The crux that proves or kills the holistic approach.
  • Judge catches the buried errors → the holistic, no-regex-extraction approach is validated; ship it.
  • Judge misses them → the suite goes red and the bet is falsified. Last-resort fallback: a deterministic per-number extraction step ahead of the judge.

This experiment is deliberate and must be explicit in the plan — it proves or kills the holistic approach before it gates. Note the mild regime in our favor: slots are short (a few sentences, ~1–4 numbers), so this is far from the long-list stress case where lost-in-the-middle is worst.

§8 CI / API-key story (stated honestly)

"Fixtures-first" freezes the eval's inputs (no proposal regeneration; a tiny, fast, cheap run) and isolates judge behaviour from generator behaviour. But the judge is itself an LLM call, so running the eval needs a model API key. This is not a "no API key needed" design.

OptionHowTrade-off
Live judge in CI recommendedANTHROPIC_API_KEY repo secret; the small frozen fixture set keeps cost/latency trivial; multi-sample with a pass threshold absorbs non-determinism.Needs a secret; a few cents of tokens per run.
Committed response cacheCommit a promptfoo response cache; CI replays recorded outputs deterministically without a key.Brittle — must be regenerated on every prompt/model change; a stale cache silently stops testing the real judge.

Recommendation: live. The fixture set is small enough that live judging is cheap and honest; the cache trades a real secret for a maintenance hazard. Either way, the eval is an offline regression layer run on prompt/model change, NOT a per-run production guard.

§9 figures_extractor schema (the ground-truth surface)

GroupFieldsSource in state
Per-fundcfs, rank, weighted_alpha, allocation_pct, per-fund exposure_pctcfs_scores, portfolio, exposure compute
Portfolio-levelweighted-avg alpha, weighted CFS, asset-class/geo exposure aggregates, count/share of funds beating benchmarkderived from portfolio + per-fund figures
Not circular. The extractor is not re-deriving figures from the workbook — that is test_proposal_validation.py's job. It re-reads the already-verified state figures and reshapes them. State is independently verified against the source workbook; the judge only asks whether the prose is consistent with those verified figures. The portfolio-level aggregates are pre-computed on purpose (§4a): they are the derivations a consultant states in prose, so surfacing them here means the judge checks membership against a ready number instead of doing the arithmetic itself.

§10 Explicitly out of scope

  • Correctness of the state figures themselves — covered by the three deterministic tiers (§2). The judge reuses verified state as trusted ground truth (see §9: not circular).
  • The fact-slot guardrail (LLM can't overwrite Python-filled slots) — already enforced structurally (keyed fragments, [UNFILLED:KEY] sentinel, check_render_fidelity).
  • Live multi-sample proposal regeneration — deferred to phase 2.
  • The qualitative-prose judge (tone/relevance/no-hallucination) — deferred.
  • RAG / retrieval evals — blocked on the regulatory-RAG workstream, no corpus yet.
  • Human-label calibration set — deferred to phase 2, not skipped. Phase 1's synthetic fixtures prove the judge catches the error types we imagined and pass hand-built good cases; they do not measure agreement with a human on real generated prose. That agreement check — a small labelled real-generation sample with a stated κ floor — is the phase-2 gate for trusting the judge beyond fixtures (see §11).

§11 Deliverable & success criteria

Success = a trustworthy prose-number entailment judge, gated in CI, over a fixture corpus that proves the judge (seeded good / seeded-bad-single / seeded-bad-buried fixtures). Concretely:

  • figures_extractor implemented + Python-unit-tested, including the pre-computed derived aggregates (§4a) so the judge does minimal arithmetic.
  • A frozen corpus across all three categories, why.*/watch.*/macro.impact, and several funds — with the good set carrying adversarial-but-correct derived cases so the suite measures precision (false-positive rate), not only recall on planted errors.
  • A Promptfoo config + rubric with a pinned Claude judge model, multi-sample repeats, and a parsed per-claim verdict (§6.3) whose reduced entailed is asserted against each fixture's expect.
  • An explicit aggregation rule over the N samples — majority-vote-of-N (or pass@k) with stated N, k, and pass threshold — not an ad-hoc average.
  • A CI job green only when every verdict matches within that threshold — the buried-error catch being the acceptance crux.
  • Reported metrics, not just pass/fail: recall on seeded-bad-* and precision on good, so a regression in either direction is visible.
Phase-2 gate (not phase 1). Before the judge is trusted beyond fixtures, a small human-labelled real-generation sample with inter-rater agreement κ ≥ ~0.6 against the judge — the accepted meta-evaluation floor. Phase 1 ships on fixtures alone; this is what earns live use.

This deliverable also feeds the eventual "published eval case study" named in docs/tasks.md.
