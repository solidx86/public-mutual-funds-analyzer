1 Summary

Re-implement the fund-consultant procedure as a runnable, headless LangGraph StateGraph in a new package consultant_engine/. Input: a client risk profile + the latest FundMaster workbook + a macro-context input. Output: a version-stamped HTML proposal that passes tests/test_proposal_validation.py.

This replaces the fund-consultant skill entirely — the skill bundle is retired once the engine exists; invocation is via CLI / Python import (no natural-language trigger). The procedure is not kept as a parallel prose spec (nothing to drift). SKILL.md decomposes into two homes, each a single source of truth:

  • Steps 1–4 (load, eligibility, filters, CFS, allocation) → Python code — precise, testable.
  • Steps 5–7 (macro narrative, jargon layering, fee presentation, HTML skeleton) → the engine's LLM prompt(s) + HTML template assets — the generation guidance becomes the prompt; the skeleton already lives in references/ and moves into the engine.

Track 0 folds in the deterministic core ENH-1/ENH-2 described — those become the spine of this engine rather than a separate accelerant.

4 Graph topology

One linear path — no mode branch, no e-Series universe restriction. Every client runs the same build.

load_profile→ load_funds +1b→ filter_universe→ score_cfs→ build_portfolio
⏸ interrupt: consultant review→resume: re-validate edits→ macro_context→ generate_proposal→ validate ⇄ repair → emit

interrupt sits on the main path, after build_portfolio — every proposal pauses for review (unless --no-review). Interaction is exit-and-resume: write the review artifact (§7) + preview, exit; state persists in the SQLite checkpointer (keyed by thread_id); --resume continues. On resume, a human-edited allocation is re-validated against the invariants before macro_context — a violating edit re-pauses (review ON) or fails loudly (--no-review).
validate → repair cycles while violations exist and iters < MAX; otherwise → emit. At cap with unresolved violations: fail loudly (non-zero exit + violation list) — never emit a broken proposal.
Only two conditionals remain: the experience-tier branch (new vs experienced, set at load_profile) at generate_proposal, and the validate→repair loop.

  • No change is required — the default file is already a valid, approvable allocation; save and --resume to accept. Edits are optional overrides.
  • Intent vs facts: the consultant edits only abbrev and allocation_pct. The computed fields (cfs, rank, risk_level, eligible) are display-only — on resume the engine re-derives them and ignores anything typed there. You cannot fake a CFS, and can only pick from the scored eligible universe.
  • Add/swap a fund: add { "abbrev": "PISTF", "allocation_pct": 15 } and rebalance to 100. On resume the engine backfills its facts and checks the eligible universe; an ineligible or over-ceiling pick produces a re-validation violation.

8 Validate → repair loop

validate recomputes from the structured ground truth in state and cross-checks the rendered HTML — the seven test_proposal_validation.py checks run programmatically: section order, CFS composite == weighted recompute (HTML slot values vs state numbers), recommended funds exist in source workbook, alpha-warning iff disqualified, retail eligibility, version stamp + AI disclosure. Emits violations[].

repair receives violations[] + current HTML and fixes only those, re-emitting HTML.

One rule module, two consumers

validate node (runtime)pytest suite (CI)

Inputthe one proposal being generatedcommitted fixtures + golden sample
LLM?yes (drives repair)no — deterministic
Jobself-correct this outputcatch code regressions

The runtime loop structurally cannot do three things the suite owns: (1) validate the validator — known-good/known-bad fixtures prove the shared module is correct; (2) gate CI — runtime validation runs at generation time and can't stop a bad commit landing, while CI pytest runs LLM-free on every push; (3) seed Track A — which re-expresses these checks as evals on the offline form. So test_proposal_validation.py is refactored, not deleted: imports the shared module, reads version from the package (not SKILL.md), points at fixtures + golden anchor.

14 Open items for writing-plans

  • Exact MAX repair iterations (default 3) + the fail-loud surface (exit code + machine-readable violation log).
  • Precise macro-contract schema fields + theme taxonomy (resolve jointly with ENH-4/ENH-6). Track 0 ships the contract + a fixture only; the live producer is ENH-6's.
  • Parity golden anchor — nominate an existing output/examples/ proposal as the structural reference the engine must reproduce at L2/L3, so "full parity" is falsifiable.
  • Fan-out/fan-in — assessed viable, but single-node is the working default. Selection interdependence (overlap dedup, diversification, satellite carving, ranks, roles) is resolved deterministically before prose, so per-fund cards are self-contained. Shape: map per-fund cards → reduce = whole-portfolio sections (Exec Summary, Macro Alignment, Strategy) + the global jargon define-once pass + HTML assembly. Refinements that shrink the map set: the two structural positions (gold PeEMAS; money market PeCDF-A, or PIMMF-A for Shariah) are always-included with formulaic rationale → template their cards deterministically, out of the LLM map set. So the map set is only the alpha-selected cards (+ optional satellite / exposure-gap) — roughly 2 in a 4-fund Starter, up to ~5 in a full Aggressive. At N≈2 fan-out isn't worth the orchestration; only marginally attractive at the high end, and then for isolation/repair + the Send-API showcase, not throughput. Caveat: naive workers break the jargon define-once-per-document rule (handle deterministically or in reduce). Decide map-vs-single at planning.
